# T300 Module Documentation

## Overview

The T300 module provides functions for calculating distances to parks (T300) using road network analysis and spatial data processing. This module is part of the 3-30-300 analysis framework for England using big spatial data, focusing on the "300 meters to a park" component of the urban forestry rule.

## Usage Examples

### Basic Workflow

The typical workflow for T300 analysis involves:

1. **Filtering spatial features** - Extracting road networks, buildings, and parks within geographic boundaries
2. **Building road network graph** - Creating a graph representation of the road network for distance calculations
3. **Calculating distances** - Computing both Manhattan (network) and Euclidean distances to parks
4. **Processing geographic areas** - Running the complete analysis for specific geographic codes

## Distance Calculation Methods

### Manhattan Distance (Network Distance)

The Manhattan distance calculation uses the road network to find the shortest path between buildings and park access points:

- **Purpose**: Represents the actual walking/driving distance along roads
- **Method**: Uses Dijkstra's algorithm to find shortest paths in the road network graph
- **Output**: `distance_manhattan` column with network distances in meters

### Euclidean Distance

The Euclidean distance calculation provides straight-line distances to park sites:

- **Purpose**: Represents the "as-the-crow-flies" distance to park boundaries
- **Method**: Uses spatial joins to find nearest park sites
- **Output**: `distance_euclidean` column with straight-line distances in meters

## Key Parameters

### Geographic Parameters

- **geo_level**: Geographic level (e.g., "LAD22CD", "MSOA21CD")
- **geo_code**: Specific geographic code to process
- **overwrite**: Whether to overwrite existing output files

### Road Network Parameters

- **road_nodes_gdf**: GeoDataFrame containing road network nodes
- **road_edges_gdf**: GeoDataFrame containing road network edges with length attributes

## Output Columns

The processed data includes:

- `verisk_premise_id`: Unique building identifier
- `closest_park_access_id`: ID of the nearest park access point
- `distance_manhattan`: Network distance to park access point (meters)
- `closest_park_site_id`: ID of the nearest park site
- `distance_euclidean`: Euclidean distance to park site (meters)

## Dependencies

This module requires:

- `geopandas` for spatial data handling
- `networkx` for graph operations and shortest path calculations
- `osmnx` for road network graph creation and distance calculations
- `pyspark` for distributed computing
- `pandas` for data manipulation
- `tqdm` for progress tracking

## Performance Considerations

- **Memory Usage**: Large road networks can consume significant memory
- **Processing Time**: Manhattan distance calculations are computationally intensive
- **Optimization**: Uses precomputed shortest paths to improve performance
- **Progress Tracking**: Includes progress bars for long-running operations

## Notes

- The module uses both Manhattan (network) and Euclidean distances for comprehensive park accessibility analysis
- Road network graphs are converted to undirected graphs for bidirectional path finding
- All distances are rounded to 1 decimal place for consistency
- The module includes comprehensive error handling and logging
- Park access points and park sites are treated separately for different types of accessibility analysis
